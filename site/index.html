The part of "AI that does things" nobody wants to build

The moment your agent needs to act as a user — send their email, open their PR, read their calendar — you're on the hook for OAuth flows, encrypted token storage, refresh races, and webhook plumbing. Get any of it wrong and you leak credentials or silently log users out.

🔒

Tokens never touch your LLM

Calls go through a proxy that injects the user's token server-side. Your agent gets a capability, not a secret.

⚡

No webhooks

Connect a user, then call provider APIs by your own user id. No async callback dance to wire up.

♻️

Refresh, handled

Expiring tokens are refreshed automatically under a lock — no double-refresh bug that logs everyone out.

Three steps. That's the integration.

No OAuth code in your app, ever.

01 · CONNECT

Get a link

Generate a connect URL for a user. They authorize the provider once.

02 · CALL

Act as them

aa.proxy(...) any provider API on that user's behalf — token injected for you.

03 · DONE

Stay connected

Tokens are vaulted, refreshed, and isolated. If a user isn't connected, you get a link back — not an error.

Built for every agent stack

TypeScript and Python SDKs with the same surface. The wedge is in the type system: if the user isn't connected, you get an AuthorizationRequiredError with a ready-to-use connect link.

agent.ts
import { Grantd, AuthorizationRequiredError } from 'grantd';
const aa = new Grantd({ apiKey });

try {
  const profile = await aa.proxy({ userId, provider: 'github', path: '/user' });
} catch (e) {
  if (e instanceof AuthorizationRequiredError) redirect(e.connectUrl);
}
from grantd import Grantd, AuthorizationRequiredError
aa = Grantd(api_key="sk_...")

try:
    profile = aa.proxy(user_id=user.id, provider="github", path="/user")
except AuthorizationRequiredError as e:
    redirect(e.connect_url)  # not connected yet

MCP-native
